# Union-Find（素集合データ構造）

Union-Find（Disjoint Set Union, DSU）は、要素の集合を「互いに素な部分集合（グループ）」に分割し、その所属関係を管理するためのデータ構造。「2つの要素は同じグループに属するか」「2つのグループを1つに統合する」という2種類の操作を高速に行うことに特化している。

このドキュメントはアルゴリズム自体の一般論。このリポジトリでの実装への当てはめは [`architecture.md`](./architecture.md) を参照。

## 基本操作

古典的には次の3操作で構成される。

| 操作 | 役割 |
| --- | --- |
| MakeSet(x) | 要素`x`だけからなる、新しい単独の集合を作る |
| Find(x) | `x`が属する集合を代表する1つの要素（root）を返す |
| Union(x, y) | `x`と`y`が属する集合を1つに統合する |

「同じ集合に属するか」（Connected/Same）は `Find(x) == Find(y)` で判定できるため、独立した操作としては定義しないことも多い。

## 素朴な実装とその問題点

各要素に「親」を1つ持たせ、根（`parent[i] == i` となる要素）をたどることでFindを実現するのが素朴な実装。Unionは「一方の根をもう一方の根の子にする」だけで済む。

ただし何も工夫しないと、Unionのやり方次第で木が横に伸びた鎖状になり、Findが最悪 `O(n)` かかる（要素数分だけ親をたどる）。

## 高速化: 経路圧縮とunion by rank/size

**経路圧縮（path compression）**: Findで根までたどった経路上の全要素の親を、直接その根に繋ぎ変える。1度Findした要素は、次回以降ほぼ `O(1)` でrootに到達できるようになる。

**union by rank / union by size**: Unionで2つの木を繋ぐ際、常に「小さい方の木の根を、大きい方の木の根の子にする」ようにする。木の高さの伸びを `O(log n)` に抑えられる（sizeは要素数、rankは木の高さの上界を基準にする方式で、どちらも狙いは同じ）。

## ならし計算量

経路圧縮とunion by rank/sizeを両方併用すると、`m`回の操作（Find/Unionの合計）にかかる時間は `O(m α(n))`。`α`はアッカーマン関数の逆関数で、現実的などんな `n` に対しても4以下の値しか取らない。実用上は「ほぼ `O(1)`」とみなしてよい。

## できないこと

古典的なUnion-Findは「一度統合した集合を分割する」操作（削除・Union解除）を直接サポートしない。一度Unionした要素同士の関係を後から切り離すには、Union操作の履歴を再生してデータ構造を作り直す、あるいはオイラーツアー木のような別のデータ構造を使う必要がある。

## 用途

- グラフの連結成分の管理（例: Kruskal法での閉路検出）
- 画像処理におけるラベリング（連結領域の抽出）
- オフラインでのグラフ連結性クエリ
