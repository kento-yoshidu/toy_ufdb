# toy_ufdb ロードマップ

Union-Find（素集合データ構造）を核にしたトイDB。Rust製、CLI(REPL)、キーのみ管理。

## バージョニング方針

メジャーバージョンごとにリポジトリを分ける。

- **v0（このリポジトリ）**: オンメモリのみ。ストレージ（永続化）は実装しない
- **v1（別リポジトリ）**: このリポジトリの設計を引き継ぎ、ストレージ（永続化）を実装する

このロードマップの「Phase」は基本的にv0（このリポジトリ）のスコープ。v1へ引き継ぐ内容は末尾の「v1への引き継ぎメモ」にまとめる。

## Phase 1: Union-Findコア (`src/lib.rs` or `src/union_find.rs`)
データ構造:
- `HashMap<String, usize>` … キー → 内部インデックス
- `Vec<usize>` parent配列
- `Vec<usize>` rank または size配列（union by rank/size用）

実装する操作:
- `make_set(&mut self, key: &str)` … 未登録キーなら新しい集合として追加。既存なら何もしない
- `find(&mut self, key: &str) -> Option<String>` … 代表元のキーを返す。path compression込み。存在しないキーは `None`
- `union(&mut self, a: &str, b: &str)` … 2つの集合を統合。存在しないキーは内部で `make_set` して扱う
- `same(&mut self, a: &str, b: &str) -> bool` … 同じ集合か判定。`UnionFind::same` と名前を揃える。`unite` と異なり、未登録キーが渡されても `make_set` はせず、単に `false` を返す（SAMEは参照系の操作であり、呼ぶだけでデータが増えるのは直感に反するため）

設計判断ポイント（実装しながら決める）:
- ~~`find` の再帰 vs ループ実装（再帰は大量データでスタックオーバーフローの可能性あり → ループ推奨）~~ → union by sizeで木の高さがO(log n)に抑えられるため、再帰のままで問題なしと結論
- union by rank と union by size のどちらにするか → **決定: size**（後の「グループ一覧・グループサイズ」機能と相性が良い）
- 存在しないキーに対する挙動 → **決定**: `unite` は自動登録（`make_set`してから統合）、`connected` は登録せず `false`

単体テストで最低限カバーすべきケース:
- 単純な union → find で同じ代表元になる
- 推移的な union（a-b, b-c → a と c が connected）
- 未登録キーの find/connected
- 同じキーを2回 union しても壊れない（冪等性）
- path compression 後も find の結果が変わらないこと

## 既知の制約（v0＝このリポジトリではやらない・やれないこと）
- **削除・分割は直接サポートしない**: 古典的Union-Findは一度unionした集合を分割する操作をサポートしない。Phase 6で「union履歴を保持しての組み直し（rebuild）」というワークアラウンドを実装する予定（詳細はPhase 6参照）。
- **永続化なし**: プロセスを終了するとデータは消える。v1（別リポジトリ）で対応する。
- **値（value）なし**: 要素はキーのみ。KVとして使いたくなったら拡張が必要（v0/v1どちらでやるかは未定、末尾の引き継ぎメモ参照）。
- **規模**: 動作確認は10万件（`N = 100,000`）規模まで。理論上はメモリが許す限り動作するはず（union by size + 経路圧縮により操作はならしO(α(n))で、キー数に対してほぼ線形にしかメモリを消費しないため）だが、それ以上の規模は未検証。

## Phase 5: DB / GROUP（2層構成にする）
`DB`（名前付きの`Ufdb`インスタンス）→ `GROUP`（各集合。代表元ごとに自動形成され、ユーザーが直接作るものではない）という2層構成にする。`USE`で切り替えるのは`DB`のみ。

- [ ] データ構造: `HashMap<String, Ufdb>`（DB名 → `Ufdb`）を持つ層を追加し、REPL側で「現在選択中のDB名」を状態として保持する
- [ ] `USE <name>` コマンド: 以降の`INSERT`/`MERGE`/`SAME`/`GROUPS`の操作対象となるDBを切り替える
- 設計判断ポイント: `USE`で存在しない名前を指定されたときに自動作成する（`make_set`と同じ自動登録方式）か、`CREATE DB`のような明示的な作成コマンドを別途用意するかを実装時に決める
- 位置づけ: この機能はv1（別リポジトリ）の永続化より先に実装する。名前付きの入れ物（DB）を先に用意しておくことで、後続の永続化は「1DB = 1ファイル」のように素直に対応させられる
- **DBは互いに完全に独立**: 各DBは別々の`Ufdb`インスタンス（独自の`keys`・独自の`UnionFind`）なので、DBをまたいだ`MERGE`/`SAME`などは成立しない。UFQLのコマンドは常に「現在`USE`で選択中の1つのDB」だけを操作対象にする。DB間でデータを行き来させたい場合は、対象DBに`USE`で切り替えた上で改めて`INSERT`/`MERGE`し直す必要がある（自動的な相互参照・インポート機構はv0では持たない）
- プロセス自体がDBより上位の概念（複数DBを保持する暗黙の入れ物）であり、これ自体に名前を付けたり切り替えたりする必要はない（MySQLサーバー自体に相当する）

## Phase 6: 削除サポート（union履歴の再生によるrebuild）
古典的Union-Findは分割・削除を直接サポートしないため、「全union操作の履歴を保持しておき、削除時は対象を除いた履歴を最初から再生してUnionFindを一から組み直す」というワークアラウンドで対応する。

- [ ] `Ufdb`（または`UnionFind`）に、これまで実行した`unite`操作の履歴を保持する仕組みを追加する（例: `Vec<(usize, usize)>`でunion順を記録）
- [ ] 削除実行時: 対象を除いた履歴を先頭から再生し、新しい`UnionFind`を一から組み直す（`add`→`unite`を履歴の順に再実行）
- コスト: 削除1回あたり **O(E・α(E))**（`E`は保持している履歴のunite操作数、`α`は経路圧縮+union by sizeでのならし計算量）を許容する。頻繁な削除がボトルネックになるようなら、より効率的な削除可能データ構造への置き換えは将来の検討課題（v1以降）
- 設計判断ポイント（実装しながら決める）:
  - 「削除」の対象は何か: 特定のキーをデータベースから完全に削除する（`DELETE <key>`）のか、特定の過去の`MERGE`操作だけを取り消す（`UNMERGE <a> <b>`的なもの）のか
  - Phase 5でDBが複数になる場合、履歴はDBごとに独立して持つ

## Phase 8: EXPLAIN（dry run）
SQLの`EXPLAIN`にならい、コマンドを実際には実行せず、内部で何が起きるかを見せる`EXPLAIN <command>`を追加する。

- [ ] `UnionFind`に経路圧縮を行わない読み取り専用の`find_readonly(&self, x: usize) -> usize`を追加する（`&self`を取ることで「状態を一切変更しない」ことを型レベルで保証する。既存の`find`は経路圧縮ありの`&mut self`のまま残す）
- [ ] `EXPLAIN SAME a b` / `EXPLAIN MERGE a b`: `find_readonly`で両者の代表元と集合サイズを求め、一致するか（`MERGE`なら union by size でどちらが親になるか、すでに同じ集合ならno-opになる旨）を表示する
- [ ] `EXPLAIN INSERT key`: 既存キーかどうか（`make_set`が`true`/`false`のどちらになるか）を表示する
- [ ] `EXPLAIN GROUPS`: 見せられる情報は少ない（全キーを走査するだけ）。実装するか自体を検討する
- 設計判断ポイント: `find_readonly`は経路圧縮をしないため、その場では通常の`find`より遅くなる（木の高さ分かかる）。EXPLAINは低頻度の診断用コマンドなので許容する

## Phase 9: STATS（統計情報）
既存のデータ（`keys`と`UnionFind`の`parent`/`size`）だけで計算できる、DB全体のサマリを表示するコマンド。どの項目を採用するかは未定（候補を列挙しておき、後で絞る）。

候補:
- 登録キー数（`N = keys.len()`）
- グループ数（代表元の種類数。`GROUPS`と同じ走査で数えられる）
- 最大/最小/平均グループサイズ（`GROUPS`の結果から集計）
- 最大木の高さ（`find_readonly`（Phase 8）を使えば副作用なく実際の深さを測れる）
- Phase 6（削除サポート）実装後なら、保持しているunion履歴の件数

- 実装方針: `GROUPS`の集計ロジックを再利用してサマリだけ出す形が自然
- 位置づけ: v0スコープ（既存のオンメモリデータからの計算のみ、ストレージ不要）

## Phase 10: コマンドごとのメモリ使用量表示（カスタムアロケータ）
実行時間の表示と対になる機能。各コマンド実行の前後で、実際に確保・解放されたバイト数を表示する。

- [ ] `#[global_allocator]` で `System` アロケータをラップした独自アロケータを定義する（`GlobalAlloc`トレートを実装）
- [ ] `alloc`/`dealloc` の中で `AtomicUsize` のカウンタを加減算し、プロセス全体のアロケーション量を追跡する
- [ ] 各コマンド実行の前後でこのカウンタを読み取り、差分をナノ秒の実行時間表示と並べて表示する
- 注意点: これは**プロセス全体**のアロケーション活動を捉える。`println!`のフォーマットや`clap`のパースなど、`Ufdb`本体とは無関係な一時アロケーションのノイズも混ざる（実行時間表示における「OSタイマー精度は実際はナノ秒より粗い」という注意書きと同様の位置づけ）
- 設計判断ポイント（実装しながら決める）:
  - 表示するのは「正味の増減（確保 − 解放）」か、「確保したバイト数の総量（解放は無視）」か。一時的なVecなどはコマンド内で確保後すぐ解放されることもあるため、正味の増減だけだとほぼ0に見えるケースがある。何を見せたいかで設計が変わる

## Phase 11: HTML出力（静的な合体グラフの可視化）
現状の`UnionFind`は経路圧縮によって`parent`が平坦化されるため、`parent`配列をそのまま辿ってもMERGEの実行履歴（どのキーとどのキーが直接繋がれたか）は復元できない。可視化用に、`UnionFind`の性能用データ（`parent`/`size`）とは別に「MERGEで指定されたキー同士の生の隣接関係」を保持するグラフを`Ufdb`に追加する。

このリポジトリ（v0）で扱うのは静的なスナップショット出力のみ。動的な更新・編集・インタラクションはTauriアプリ側（別リポジトリ）の責務とし、ここでは実装しない。

- [ ] `Ufdb`に`graph: HashMap<String, Vec<String>>`（キー→隣接キー一覧）を追加する。無向グラフとして両方向（a→b, b→a）を記録する
- [ ] `unite(key_a, key_b)`が呼ばれるたびに、戻り値（union by sizeで実際に集合が変わったか）に関わらず`graph`にa-bの辺を追加する。これは「実際に呼ばれたMERGEの記録」であり、`UnionFind`内部の集合構造（経路圧縮で変化する`parent`）とは独立させる
- [ ] `EXPORT HTML`（仮称。コマンド名は実装時に決める）: 現在の`graph`・`GROUPS`の状態をもとに静的なHTML文字列を組み立てる
- **出力方式: サーバー起動+ブラウザ自動オープン（Storybook的な体験）**
  - `EXPORT HTML`はstdoutには出さず、常にローカルサーバーを起動してHTMLを配信し、`open`クレートでデフォルトブラウザにそのURLを開かせる
  - サーバーの生存期間は**1リクエスト分だけ**: GETリクエストを1回受けてHTMLを返したらリスナーを閉じ、REPLに制御を戻す（バックグラウンドで生かし続けない。マルチスレッドや`Ufdb`の共有状態管理が不要になり実装がシンプル）。最新化したい場合は改めて`EXPORT HTML`を呼び直す運用にする
  - 実装は`tiny_http`（軽量な同期HTTPサーバー、asyncランタイム不要）+ `open`（クロスプラットフォームでブラウザ起動）を使う。生の`TcpListener`でHTTPを手書きする案やactix/axumのようなフルフレームワークは見送る
  - これはPhase 7のCSV export（標準出力のみ、ファイルI/Oなし）とは異なる出力経路になるが、ディスクに何も書き込まない・プロセス終了とともに消える一時的なTCPリスナーである点でv0の「永続化なし」方針とは矛盾しない
  - 新規依存として`Cargo.toml`に`tiny_http`・`open`を追加する
- **非機能要件: 大規模GROUPへの対応方針**
  - 1万ノードのGROUPをそのまま1枚のグラフとして忠実に再現するのは非現実的（技術的な描画コストというより、人間が読み取れる情報量を超えるという可読性の限界が本質）
  - 判定は**GROUPごと**に行う（DB全体の合計ノード数ではない）。小さいGROUPが大量にあってもそれぞれ普通に描画され、閾値を超える巨大GROUPだけが個別に省略対象になる
  - 閾値を超えたGROUPは、グラフ描画（レイアウト計算）を諦める。デフォルトでは`<details>`/`<summary>`要素（HTML標準機能、JS不要）で「○ GROUP 3: 12,483 keys」のように折りたたんだサイズ表示にしておき、クリックで展開できるようにする
  - 展開後の中身は**フルグラフではなく軽量な代替表示**（メンバーキーの単純な一覧）に留める。巨大GROUPに対してレイアウト計算そのものを行わないという方針は維持し、export時の計算コスト・HTMLサイズを抑える
  - 閾値以下のGROUPは最初から展開された状態でグラフ描画する。閾値の具体的な値、および描画方式（DOM/SVG/canvas）は実装時に決める
- 設計判断ポイント（実装しながら決める）:
  - 同じ`a b`に対して複数回`MERGE`された場合、`graph`に辺を重複して持たせるか、重複排除するか
  - HTML側の描画方法（DOM構造だけで表現するか、簡易的なSVGや`<canvas>`スクリプトを埋め込むか）
- Phase 6（削除のためのunion履歴）とはあえて分離する: Phase 6の履歴は「rebuildのための操作列」、この`graph`は「現在の合体関係を表示するための隣接リスト」であり目的が異なるため、別々のデータとして持つ（実装時、`unite`内で2箇所更新することになる点に注意）

## 検討事項（未定）: オイラーツアー木（ETT）による高速な削除
Phase 6のrebuild方式（O(E・α(E))）の代わりに、削除を**O(log n)**まで高速化できる案。森全体を1本のオイラーツアー系列として平衡二分探索木（splay木など）で持ち、`unite`（link）/削除（cut）/`same`（同じ木か判定）をすべてO(log n)で行う。

- 前提: 今の`unite`は同じ集合ならno-op（辺を作らない）ので、森に冗長な辺（サイクル）は発生しない。この前提がある限り、素のETTで「特定のMERGEの取り消し＝O(log n)」が実現できる
- 「辺を切った後も別の辺経由でまだ繋がっているか」まで扱う完全動的連結性が必要になった場合は、ETT＋辺のレベル分け（Holm–de Lichtenberg–Thorup算法など）が必要になり、コストもO(log²n)ならしまで上がる。より複雑
- `parent`/`size`の`Vec`2本というシンプルな今の実装を、平衡二分探索木ベースの実装に丸ごと置き換える必要があり、実装コストはPhase 6よりかなり大きい
- 位置づけ: Phase 1やPhase 6には組み込まず、あくまで学習用の検討事項として保留。着手するかは未定

## 検討事項（未定・優先度低）: TCPサーバー化
CLIのREPLプロセスと、別プロセス（Tauriなど）とで同じ「生きているデータ」を共有したくなった場合にのみ必要になる話。プロセスを跨いでメモリを共有することはできないため、どちらか一方のプロセスがデータの実体を持ち、もう一方はネットワーク経由でそれを操作するクライアントになる構成が要る。その場合`Ufdb`への並行アクセス制御（`Arc<Mutex<Ufdb>>`など）も新たに必要になり、単一プロセス前提の今の実装より一段大きな作業になる。

現時点ではTauri側もCLIとデータを共有する必要はなく（下記メモ参照）、着手の予定はない。方針が変わったら改めて計画する。
